Asymmetric Split: One Column Carries the Weight, the Other Runs Beside It

Two columns run the full length of the page, but never as equals — the body claims two-thirds of the width and all of the argument; the rail claims one-third and carries everything that supports it without ever interrupting it.

The concept is arithmetic before it's anything else: eight columns of a twelve-column grid go to the body, four go to the rail. That 2:1 ratio is fixed and never approaches parity — the moment a rail widens toward half the page, it stops reading as marginalia and starts reading as a second, competing column. Imbalance is the entire structuring device here; there's no lead image, no size-varied card grid, no color coding to lean on instead.

"The rail isn't a smaller version of the body. It's a different kind of writing — running commentary, not a second draft of the argument — and its width should say so before a reader reads a word of it."

Use this when you have one continuous argument that benefits from a persistent, parallel channel: an essay with citations that should stay visible rather than jump to endnotes, a long article with a running "related coverage" rail, an internal report where a reviewer's marginal commentary needs to sit beside the text it responds to, page after page — not just wherever a footnote mark happens to fall.

Reading This Archetype

When Not to Use It

If every note needs to anchor to one specific sentence — not just the general vicinity of a section — reach for tufte-sidenote instead, which pins each note to its triggering paragraph at the same vertical position. This rail runs continuously and independently; it's built for commentary that accompanies a whole passage, not commentary that must line up with a single line.

The Structural Rule

Eight columns to four, always. Never widen the rail chasing "more room" for a note — if content genuinely needs more space, it belongs in the body as a subsection, not in a stretched rail. Drifting toward 50/50 is the one failure mode that undoes this archetype.

Editorial Devices It Favors

One pull quote breaking the body column, at its most quotable line. Numbered, hairline-ruled rail entries — never boxed, never shadowed. Citations and related links stay text-only in the rail; the body is the only region that ever earns a captioned exhibit.

Why the Migration Took Six Months Longer Than the Postmortem Admits

The public timeline said four months. Internally, everyone counted from a different start date, because the actual decision to move off the old queueing system was made eleven weeks before anyone wrote it into a roadmap. Those eleven weeks don't appear in any postmortem, but they explain most of what went wrong later — three separate teams had already built workarounds for problems the migration was supposed to fix, and none of those workarounds were listed as dependencies when the project finally got a start date.

"A project doesn't start when it gets a ticket number. It starts when the first workaround gets written — and that's the date nobody puts in the postmortem."

Once the workarounds surfaced, the plan doubled in scope without anyone officially re-scoping it. The team kept the original four-month estimate in every status update because renegotiating it felt like admitting the first estimate was wrong, which it was, for reasons that had nothing to do with anyone's execution.

What finally changed the timeline publicly wasn't a new estimate — it was a one-line addition to the weekly update: "scope grew by three undocumented dependencies in week one." Nobody had said "undocumented" out loud before that. The rest of the conversation got easier once the word existed.

Review Latency Rose Even as Headcount on the Team Grew

Headcount is up 40% since Q1, and median time-to-first-review has gone from 6 hours to 14. That divergence is the finding, not the raw latency number — more reviewers should have made this number go down, and instead it nearly doubled, which points at a coordination problem, not a capacity one.

The likely cause: reviewer assignment is still round-robin across the whole team rather than routed by familiarity with the code area, so newer reviewers are frequently assigned PRs they need extra time to evaluate — time that shows up in the aggregate median even though any individual review isn't taking unusually long.

Rate Limits: How the API Enforces Them and What You'll See

Every endpoint is limited per API key, not per IP — 600 requests per minute on standard keys, 3,000 on enterprise keys, measured on a rolling one-minute window rather than a fixed clock-aligned bucket. That means a burst of 600 requests at 0:59 and another 600 at 1:01 will both succeed; a burst of 1,200 requests inside any single rolling minute will not.

When a request is rejected, the response is a 429 with a Retry-After header in seconds — always trust that header over any client-side backoff timer you've hardcoded, since the window's start time depends on your own recent request pattern, not a fixed schedule.
